# i18n (English + Croatian) Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add English + Croatian language support to KoachApp, with a language switcher on the homepage, in the client profile/settings page, and in the coach sidebar. Croatian remains the default for existing users; English-speaking visitors get English automatically on first visit.

**Architecture:** Use `next-intl` in its cookie-based (non-routing) mode so existing URLs are unchanged. Resolve locale per request from a helper that reads (1) `profiles.language` for logged-in users, (2) `NEXT_LOCALE` cookie, (3) `Accept-Language` header, defaulting to `hr`. All UI strings move to `messages/hr.json` and `messages/en.json`. The 60 pre-seeded foods get a second column `name_en`; all coach-created content stays in whatever language it was typed.

**Tech Stack:** Next.js 16 App Router (webpack), React 19, Supabase SSR, Tailwind v4, `next-intl` (new), TypeScript.

**Design doc:** [docs/plans/2026-04-24-i18n-design.md](../plans/2026-04-24-i18n-design.md)

**Important context:**
- This codebase has **no test framework**. Verification is via `pnpm build` / `next build --webpack`, `pnpm lint`, and the `preview_*` MCP tools (start dev server + snapshot + console logs).
- Next.js 16 has breaking changes. The in-repo i18n guide lives at `node_modules/next/dist/docs/01-app/02-guides/internationalization.md` and recommends libraries like `next-intl`. We're using next-intl's **cookie-based** flow (not routing-based) — confirmed docs reference.
- Migrations live in `supabase/migrations/` with `YYYYMMDD_name.sql` naming. Apply via Supabase MCP `apply_migration` AND keep the SQL file in the repo for history.
- Coach sidebar lives at `app/coach/layout.tsx` — no existing "settings" page for the coach, so the language switcher goes in the sidebar directly.
- Client settings live inside `app/app/profile/page.tsx`.

**Step 2:** Create `i18n/request.ts` with the server-side locale resolver.

```ts
// i18n/request.ts
import { getRequestConfig } from "next-intl/server";
import { cookies, headers } from "next/headers";
import { createServerClient } from "@supabase/ssr";

export const SUPPORTED_LOCALES = ["hr", "en"] as const;
export type Locale = (typeof SUPPORTED_LOCALES)[number];
export const DEFAULT_LOCALE: Locale = "hr";

function isLocale(v: string | undefined | null): v is Locale {
  return v === "hr" || v === "en";
}

function pickFromAcceptLanguage(header: string | null): Locale {
  if (!header) return DEFAULT_LOCALE;
  // Very small parser: first match wins.
  const parts = header.split(",").map((p) => p.trim().split(";")[0].toLowerCase());
  for (const p of parts) {
    if (p.startsWith("en")) return "en";
    if (p.startsWith("hr")) return "hr";
  }
  return DEFAULT_LOCALE;
}

export async function resolveLocale(): Promise<Locale> {
  // 1. Logged-in user: profiles.language wins.
  try {
    const cookieStore = await cookies();
    const supabase = createServerClient(
      process.env.NEXT_PUBLIC_SUPABASE_URL!,
      process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
      {
        cookies: {
          getAll: () => cookieStore.getAll(),
          setAll: () => {},
        },
      }
    );
    const { data: { user } } = await supabase.auth.getUser();
    if (user) {
      const { data: profile } = await supabase
        .from("profiles")
        .select("language")
        .eq("id", user.id)
        .single();
      if (isLocale(profile?.language)) return profile.language;
    }
  } catch {
    // fall through
  }

  // 2. Cookie override (also used pre-login).
  const ck = (await cookies()).get("NEXT_LOCALE")?.value;
  if (isLocale(ck)) return ck;

  // 3. Accept-Language header.
  const h = await headers();
  return pickFromAcceptLanguage(h.get("accept-language"));
}

export default getRequestConfig(async () => {
  const locale = await resolveLocale();
  const messages = (await import(`../messages/${locale}.json`)).default;
  return { locale, messages };
});
```

### Task 0.5: Server action to set language

**Files:**
- Create: `app/actions/set-language.ts`

**Step 1:**

```ts
"use server";

import { cookies } from "next/headers";
import { createServerClient } from "@supabase/ssr";
import { revalidatePath } from "next/cache";
import { SUPPORTED_LOCALES, type Locale } from "@/i18n/request";

export async function setLanguage(locale: Locale) {
  if (!SUPPORTED_LOCALES.includes(locale)) throw new Error("Invalid locale");

  const cookieStore = await cookies();
  cookieStore.set("NEXT_LOCALE", locale, {
    path: "/",
    maxAge: 60 * 60 * 24 * 365,
    sameSite: "lax",
  });

  // If user is logged in, also persist to profile.
  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll: () => cookieStore.getAll(),
        setAll: (list) =>
          list.forEach(({ name, value, options }) =>
            cookieStore.set(name, value, options)
          ),
      },
    }
  );
  const { data: { user } } = await supabase.auth.getUser();
  if (user) {
    await supabase.from("profiles").update({ language: locale }).eq("id", user.id);
  }

  revalidatePath("/", "layout");
}
```

### Task 0.6: LanguageSwitcher component

**Files:**
- Create: `components/language-switcher.tsx`

**Step 1:**

```tsx
"use client";

import { useLocale } from "next-intl";
import { useTransition } from "react";
import { setLanguage } from "@/app/actions/set-language";

export function LanguageSwitcher({ className }: { className?: string }) {
  const locale = useLocale();
  const [pending, startTransition] = useTransition();

  function change(next: "hr" | "en") {
    if (next === locale) return;
    startTransition(() => setLanguage(next));
  }

  return (
    <div className={`inline-flex gap-1 ${className ?? ""}`} role="group" aria-label="Language">
      {(["hr", "en"] as const).map((code) => (
        <button
          key={code}
          type="button"
          disabled={pending}
          onClick={() => change(code)}
          className={`px-2 py-1 text-xs rounded border ${
            locale === code ? "bg-foreground text-background" : "bg-background"
          }`}
        >
          {code.toUpperCase()}
        </button>
      ))}
    </div>
  );
}
```

## Phase 1 — Translate UI, page by page

**Pattern for each page task (same shape every time):**

1. Open the page file and its immediate client-component children (the `*-manager.tsx`, `*-builder.tsx`, inline form components).
2. List every hardcoded Croatian user-visible string (button labels, placeholders, headings, toasts, empty states, validation messages, aria-labels).
3. Invent a namespaced key per string (e.g. `coach.clients.new.submitButton`).
4. Add the Croatian value under that key in `messages/hr.json` and the English translation in `messages/en.json`. Keep keys sorted by namespace.
5. Replace inline strings in JSX:
   - Server component: `import { getTranslations } from "next-intl/server"; const t = await getTranslations("coach.clients.new");`
   - Client component: `import { useTranslations } from "next-intl"; const t = useTranslations("coach.clients.new");`
   - Then `t("submitButton")` in place of "Kreiraj klijenta".
6. For toast messages / validation messages used inside `"use server"` actions: these CAN'T use `useTranslations`. Translate them using `getTranslations()` at the top of the action, OR pass through a translated error code string and translate in the UI. Prefer the second pattern (return `{ error: "email_already_exists" }` and translate on the client) to keep server actions locale-agnostic.
7. Verify:
   - `pnpm build` — clean.
   - `preview_start` if not running.
   - Load the page; snapshot. Toggle language via cookie (`preview_eval: document.cookie='NEXT_LOCALE=en; path=/'; window.location.reload()`) or via switcher once placed. Snapshot in EN.
   - Confirm: no Croatian left when EN, no English left when HR.
8. Commit with message `i18n: translate <page>`.

Shared namespaces (define once, reuse):
- `common.*` — Save, Cancel, Delete, Edit, Close, Loading, Error, Required, etc.
- `auth.*` — email, password, login, logout, etc.
- `validation.*` — generic form validation messages.

## Notes on pitfalls

- **Server actions calling `t()`**: `next-intl`'s `getTranslations()` works in server actions too, but prefer returning error codes from actions and translating them on the client side. This keeps actions locale-agnostic and makes them easier to test.
- **Toast messages (`sonner`)**: Toasts are triggered client-side, so `useTranslations()` works. Pass translated strings to `toast.success(t('...'))`.
- **Email content** (invite email, password reset) is out of scope — those are Supabase system emails. Igor can configure those templates separately in the Supabase dashboard if needed.
- **Dates and numbers**: use `Intl.DateTimeFormat`/`Intl.NumberFormat` with the resolved locale, not hardcoded formatting.
- **`html lang` attribute**: already handled in Task 0.4 — important for accessibility + SEO.
- **PWA cache**: next-pwa caches aggressively. After deploy, users may need to hard-reload once to pick up the language switcher. If stale translations become a problem, bust the service worker cache via version bump.
