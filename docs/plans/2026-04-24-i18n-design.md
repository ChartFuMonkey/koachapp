# i18n — English + Croatian — Design

**Date:** 2026-04-24
**Status:** Approved, awaiting implementation plan

## Context

KoachApp currently ships as Croatian-only. Every UI string is hardcoded Croatian in JSX across ~20 pages (client app under `/app/*`, coach app under `/coach/*`, landing, login, set-password). The 60 pre-seeded foods in the `foods` table are also Croatian. There is no i18n library installed and no `language` preference stored anywhere.

We want English-speaking users (clients or coaches) to have a fully localized experience, and Croatian users to keep their current experience. Users need to switch languages on the public homepage as well as in their settings (both client and coach).

## Decisions made during brainstorming

1. **Coach-created content stays in the language it was entered.** Exercises, meals, and foods a coach creates going forward do not get auto-translated. An English client may see a Croatian meal name if their coach typed it in Croatian — this is acceptable.
2. **The 60 pre-seeded foods get proper English names added.** One-time manual translation.
3. **First-visit default = browser language.** English browser → English; anything else → Croatian. User can always override via the switcher.
4. **Cookie-based locale, no URL changes.** Routes stay as-is (`/app/workout`, `/coach/clients`, etc.). Locale travels via a `NEXT_LOCALE` cookie and, for logged-in users, the `profiles.language` column (DB is the source of truth when logged in; cookie is the fallback for anonymous visitors).

## Approach

### Library
Use **next-intl** — the standard i18n library for Next.js 16 App Router. Supports both server and client components, which the codebase mixes heavily.

### Translation files
- `messages/hr.json` — existing Croatian strings, extracted as they are from the JSX today
- `messages/en.json` — English translations of the same keys

Keys organized by page/feature, e.g. `app.workout.startButton`, `coach.clients.empty`, `common.save`.

### Language resolution
Order of precedence when rendering a page:

1. If user is logged in → `profiles.language` (synced across devices)
2. Else if `NEXT_LOCALE` cookie is set → use that
3. Else → detect from `Accept-Language` header, fall back to `hr`

This is computed once in a helper used by the root layout. The middleware is not involved in locale logic (keeps the auth middleware simple).

### DB schema changes
One migration:

```sql
alter table profiles add column language text not null default 'hr' check (language in ('hr','en'));
alter table foods    add column name_en text;
```

Then a data migration that fills `foods.name_en` for the 60 pre-seeded foods.

Food display logic: `select ... coalesce(case when lang='en' then name_en end, name) as display_name ...`. If a food's `name_en` is null (coach-created food), we fall back to `name`.

### Language switcher
A small client component with two buttons (HR / EN). Placed in three spots:

- Landing page header (`app/page.tsx`)
- Client settings page (`app/app/profile/page.tsx` or similar)
- Coach settings page (need to identify — likely a sub-page of `/coach`)

When toggled:
- If logged in: updates `profiles.language` via a server action + updates the cookie
- If anonymous: updates the cookie only
- Triggers a router refresh so the new language renders immediately

## Files affected

**New files**
- `messages/hr.json`, `messages/en.json`
- `i18n/config.ts` (locale constants + resolver)
- `i18n/request.ts` (next-intl server config)
- `components/LanguageSwitcher.tsx`
- A migration file under `supabase/migrations/`

**Modified files**
- `app/layout.tsx` — wrap in `NextIntlClientProvider`, resolve locale
- `next.config.ts` (or `.mjs`) — add next-intl plugin
- Every page under `app/`, `app/app/`, `app/coach/` — replace hardcoded Croatian with `t('key')` calls (~20 pages, ~150-200 strings total)
- `middleware.ts` — no change needed
- Anywhere foods/meals/exercises are rendered — read user's language from session helper to pick `name` vs `name_en` for pre-seeded foods

## Verification

End-to-end check after implementation:

1. Log in as coach, open settings, flip language to EN. Reload. All coach UI renders in English. Flip back to HR, all renders in Croatian.
2. Log in as client, same test.
3. Log out. Visit `/`. Landing page respects browser `Accept-Language` on first visit. Use switcher in header — page re-renders in chosen language. Refresh — language persists via cookie.
4. Log in after switching to EN anonymously — session takes over, and the user's saved `profiles.language` now drives the language.
5. Open meal builder as English user: pre-seeded foods show in English (e.g. "Chicken breast"), coach-created foods show in whatever language coach typed.
6. Confirm no Croatian strings remain in EN mode by grepping for known Croatian words (Prijava, Spremi, Odustani, etc.) with the app rendered in English.

## Out of scope (for this change)

- Auto-translation of coach-created content
- Language-specific date/number formatting beyond what next-intl gives for free
- RTL languages
- Additional languages beyond HR + EN
- Translating Supabase system emails (invite, password reset)

## Scope reality

Mechanically large but low-risk: no functional logic changes, just routing text through a translation layer. Expected as a single large PR. Plan for per-page review because it's easy to miss strings.
